| 机制            | 主要作用           | 生命周期             | 典型内容                                                 |
| :-------------- | :----------------- | :------------------- | :------------------------------------------------------- |
| Memory          | 跨会话持久化记忆   | 多次会话之间长期保留 | 用户偏好、项目背景、稳定工作流、外部资源指针             |
| Context Compact | 当前会话上下文压缩 | 当前会话内生效       | 已完成的对话、工具调用结果、文件修改记录、错误与修复过程 |

因此，本文讨论的 **Context 压缩** 更接近“会话摘要与上下文替换”，它的核心目标是：在不超过模型上下文窗口的前提下，让后续模型仍能继续当前任务。

在长任务中，Agent Harness 会不断积累：

1. 用户的多轮需求变化；
2. 助手的分析过程和执行动作；
3. 工具调用结果；
4. 文件路径、代码片段、错误日志；
5. 中间产物和用户反馈；
6. 多媒体输入，例如图片、PDF 截图等。

如果全部保留，会带来几个问题：

- **上下文窗口溢出**：服务端直接报 context 超限；
- **推理成本增加**：无关历史越多，模型读上下文越慢、成本越高；
- **注意力稀释**：模型可能被旧任务或无关工具结果干扰；
- **响应延迟变高**：每次请求都要携带大量历史；
- **压缩时机不当**：如果临近超限时才压缩，用户会明显感受到等待。

# Claude Code

## Claude Code 的四类 Context 压缩方式

Claude Code 在已发布代码中主要包含四种 Context 压缩方式。

| 压缩方式           | 触发逻辑                          | 主要作用                                     | 当前状态               |
| :----------------- | :-------------------------------- | :------------------------------------------- | :--------------------- |
| microcompact       | 工具调用结果超过一定时间后清理    | 减少旧 tool result 对上下文的占用            | 默认关闭               |
| autoCompact        | 上下文达到规则阈值后主动触发      | 在超限前主动生成摘要并替换历史上下文         | 常规核心机制           |
| reactiveCompact    | 服务端返回 context 超限错误后触发 | 作为兜底机制，对较短历史压缩并保留最近上下文 | 超限恢复机制           |
| precomputedCompact | 达到阈值后后台提前压缩            | 降低用户等待压缩的时间                       | 默认关闭，受服务端控制 |

其中，**autoCompact** 是最常见的主动压缩方式，负责在上下文接近限制时提前处理。**reactiveCompact** 则是兜底方案，当服务端已经报出 context 超限时，再尝试压缩更短历史并保留最近未压缩内容。对于较大的多媒体内容，reactiveCompact 还会考虑裁剪，以进一步降低上下文占用。

**microcompact** 更偏向轻量清理，它主要处理过去工具调用结果的冗余问题。材料中提到，它的时间阈值考虑了服务端缓存有效时间，目前设置为 60 分钟，但默认关闭。

**precomputedCompact** 则是体验优化方案。它会在后台提前生成压缩结果，等真正需要替换 context 时可以直接使用，从而减少用户等待。不过该能力默认关闭，并且受服务端控制。

这四种方式对应了不同层次的问题：microcompact 处理冗余工具结果，autoCompact 处理常规长上下文，reactiveCompact 处理超限错误，precomputedCompact 处理用户等待时间。

## Summary Prompt 的设计重点

```markdown
关键：只用纯文本回复。不要调用任何工具。

- 不要使用 Read、Bash、Grep、Glob、Edit、Write 或任何其他工具。
- 你在上方对话里已经拥有所需的全部上下文。
- 工具调用会被拒绝，并浪费掉你唯一的一轮——你将无法完成任务。
- 你的整个回复必须是纯文本：一个 <analysis> 块，后面跟一个 <summary> 块。

你的任务是为目前为止的对话创建一份详细摘要，密切关注用户的显式请求以及你此前的动作。
这份摘要应当全面捕捉技术细节、代码模式以及架构决策——这些对于在不丢失上下文的前提下继续开发工作至关重要。

在给出最终摘要之前，把你的分析包裹在 <analysis> 标签里，以梳理思路并确保你已覆盖所有必要的点。在你的分析过程中：
1. 按时间顺序分析对话的每一条消息和每一段。对每一段都彻底识别出：
- 用户的显式请求与意图
- 你应对用户请求所采取的方法
- 关键决策、技术概念和代码模式
- 具体细节，例如：
- 文件名
- 完整的代码片段
- 函数签名
- 文件改动
- 你遇到的错误以及你如何修复它们
- 特别留意你收到的具体用户反馈，尤其是当用户要求你换一种做法时。
2. 复核技术准确性与完整性，彻底处理每一项要求的要素。

你的摘要应包含以下各节：
1. 主要请求与意图（Primary Request and Intent）：详细捕捉用户的所有显式请求与意图。
2. 关键技术概念（Key Technical Concepts）：列出讨论过的所有重要技术概念、技术与框架。
3. 文件与代码段（Files and Code Sections）：枚举查看、修改或创建过的具体文件和代码段。特别留意最近的消息，在适用处包含完整代码片段，并附上一段说明该次文件读取或改动为何重要。
4. 错误与修复（Errors and fixes）：列出你遇到的所有错误，以及你如何修复它们。特别留意你收到的具体用户反馈，尤其是当用户要求你换一种做法时。
5. 问题求解（Problem Solving）：记录已解决的问题以及任何进行中的排障工作。
6. 所有用户消息（All user messages）：列出所有不是工具结果的用户消息。它们对于理解用户的反馈和意图变化至关重要。
7. 待办任务（Pending Tasks）：概述你被显式要求处理的任何待办任务。
8. 当前工作（Current Work）：详尽描述紧接这次摘要请求之前正在进行的具体工作，特别留意用户和助手双方最近的消息。在适用处包含文件名和代码片段。
9. 可选的下一步（Optional Next Step）：列出与你最近所做工作相关的下一步。重要：确保这一步与用户最近的显式请求、以及你紧接这次摘要请求之前正在进行的任务直接一致。如果你上一个任务已经收尾，那么只在下一步显式契合用户请求时才列出。不要在未与用户确认的情况下，去做跑题的请求或那些早已完成的很旧的请求。
如果有下一步，请包含来自最近对话的直接引语，准确显示你当时在做什么任务、停在了哪里。这部分应逐字照录，以确保对任务的理解不发生漂移。

提醒：不要调用任何工具。只用纯文本回应——一个 <analysis> 块后跟一个 <summary> 块。任何工具调用都将被拒绝，你将因此失败本任务。
```

Claude Code 的 Summary Prompt 采用“两段式”结构，要求先输出 `<analysis>`，再输出 `<summary>`。这是一种典型的压缩前自我梳理设计，目的是让模型先系统分析对话历史，再生成可用于继续任务的摘要。

Prompt 开头和结尾都反复强调：**不得调用任何工具，只能基于已有对话内容生成纯文本摘要**。这一点非常关键，因为 Context 压缩阶段的目标是压缩已有上下文，而不是重新读取文件、执行命令或继续探索。如果在压缩阶段允许工具调用，反而会增加上下文、浪费压缩回合，并可能使模型偏离摘要任务。

Summary Prompt 要求摘要覆盖九类内容：

| 模块                       | 作用                                 |
| :------------------------- | :----------------------------------- |
| Primary Request and Intent | 保留用户的主要请求和真实意图         |
| Key Technical Concepts     | 保留关键技术概念、框架和方法         |
| Files and Code Sections    | 保留涉及的文件、代码段、函数和改动   |
| Errors and fixes           | 记录错误以及对应修复方式             |
| Problem Solving            | 记录已解决问题和仍在排查的问题       |
| All user messages          | 保留用户原始消息，防止需求和反馈丢失 |
| Pending Tasks              | 记录尚未完成的显式任务               |
| Current Work               | 记录压缩前正在进行的具体工作         |
| Optional Next Step         | 给出与最近任务直接相关的下一步       |

这说明 Claude Code 的 summary 并不是普通的对话总结，而是面向后续开发的“任务恢复摘要”。它需要让下一个模型清楚知道：用户要什么、做过什么、改过哪里、遇到什么错误、用户纠正过什么、当前停在哪里、下一步应该做什么。

其中最值得注意的是 **All user messages** 和 **Errors and fixes**。前者用于保留用户反馈和意图变化，后者用于避免后续模型重复踩坑。对于长任务来说，用户的否定、纠正和补充约束往往比普通技术细节更重要。

## 局部压缩

除了默认的全局压缩，Claude Code 还支持两种局部压缩方式，主要用于 rewind 场景，需要用户在历史对话中指定某个 user message 位置。

**Summarize up to here** 比较符合直觉，它表示把某个位置之前的历史上下文压缩掉，保留该位置之后的内容。适合早期讨论已经完成，但后续任务仍要继续的情况。

可以理解为：

```text
旧历史上下文 → 压缩为摘要
指定位置之后的上下文 → 保持完整
```

**Summarize from here** 更有 Agent 使用范式上的意义。它适合在发现 AI 后续方向跑偏、某个尝试失败或用户想回退到历史节点时使用。系统可以把该位置之后的尝试压缩掉，但保留其经验和教训，然后从指定位置继续尝试其他方向。

可以理解为：

```text
保留回退点之前的上下文
把回退点之后的失败尝试压缩为摘要
继续探索新的方向
```

这种机制与 SubAgent 隔离 context 的思想类似：**都不是让所有探索过程污染主上下文，而是把探索结果以更可控的方式保留下来。**进一步扩展后，它可以支持类似“树形 Context”的多路径探索：不同方案在不同分支中尝试，最后只把有效结论汇总回主上下文。

# Codex 与 OpenAI Agents SDK Context

Codex 的 Context 压缩机制与 Claude Code 明显不同。Claude Code 更偏向在本地 Agent Harness 中完成压缩、生成可见 summary，并通过多种策略处理上下文超限；而 **Codex 的压缩主要依赖 OpenAI/Azure 服务端完成，本地只负责判断是否需要触发压缩**。

因此，Codex 的 Context 压缩更像是一种 **服务端托管的上下文检查点机制**。它不强调本地对历史上下文进行复杂裁剪和重写，而是把压缩能力交给服务端模型或专用 endpoint，并在后续请求中使用压缩后的上下文状态继续任务。

## Codex 的三种 Context 压缩路径

Codex 目前包含三种 Context 压缩实现方式。

| 压缩方式    | 实现位置                | 主要用途            | 特点                                |
| :---------- | :---------------------- | :------------------ | :---------------------------------- |
| 本地 inline | 本地执行                | 适配非 OpenAI 模型  | 传统本地摘要压缩方式                |
| 远程 v1     | 服务端独立 endpoint     | OpenAI 模型默认路径 | 调用专门的 Context 压缩接口         |
| 远程 v2     | 服务端 session 末尾请求 | 新一代远程压缩方案  | 通过在 session 末尾追加压缩请求触发 |

实际使用中，OpenAI 模型主要走远程压缩模式，目前默认是 **远程 v1**。这意味着 Codex 的 Context 压缩并不是在本地直接生成一段明文 summary 后塞回上下文，而是通过服务端机制完成压缩与状态替换。

本地 inline 路径主要用于兼容非 OpenAI 模型。当无法使用 OpenAI 服务端压缩能力时，Codex 才需要在本地生成可读的交接摘要。

## 本地 inline 压缩 Prompt 的作用

```markdown
你正在执行一次"上下文检查点压缩"（CONTEXT CHECKPOINT COMPACTION）。为将要接手该任务的另一个 LLM 创建一份交接摘要。

应包含：
- 当前进展以及已做出的关键决策
- 重要的上下文、约束或用户偏好
- 还有哪些事情没做完（清晰的下一步）
- 继续工作所需的任何关键数据、示例或引用

保持简洁、结构化，并聚焦于帮助下一个 LLM 无缝地继续工作。
```

Prompt 要求摘要包含四类信息：

| 内容                       | 作用                         |
| :------------------------- | :--------------------------- |
| 当前进展与关键决策         | 告诉后续模型任务已经做到哪里 |
| 重要上下文、约束或用户偏好 | 避免后续模型忽略用户要求     |
| 剩余待办与下一步           | 让任务可以继续推进           |
| 关键数据、示例或引用       | 保留继续任务所需的必要证据   |

这个 Prompt 的设计目标比较明确：不是总结完整对话，而是生成一个 **任务交接摘要**。它强调简洁、结构化，并聚焦于帮助下一个模型无缝继续工作。

## 压缩后的提示方式

Codex 在压缩后，会用一段提示告诉后续模型：

> 另一个语言模型已经开始解决该问题，并生成了其思考过程摘要。你可以访问该模型使用过的工具状态，应基于已有工作继续推进，避免重复劳动。

这段提示体现出 Codex 对 Context 压缩的定位：

```text
前一个模型的工作过程
        ↓
压缩为 summary / encrypted state
        ↓
后一个模型读取该状态
        ↓
继续任务，避免重复劳动
```

也就是说，压缩后的上下文不是单纯作为“历史记录”，而是作为后续模型的 **handoff state**。它的重点是任务连续性，而不是完整复盘。

## 服务端压缩与 encrypted_content

Codex 与 OpenAI Agents SDK 的一个关键差异是：远程压缩不会返回明文 summary，而是返回类似 `encrypted_content` 的内容。

这与加密 thinking 的设计类似。用户和本地 Harness 不一定能直接看到压缩后的真实摘要文本，但服务端可以在后续请求中使用这部分加密内容恢复上下文状态。

这种设计带来几个影响：

| 方面     | 分析                                               |
| :------- | :------------------------------------------------- |
| 可观察性 | 用户无法直接检查完整压缩摘要                       |
| 安全性   | 内部压缩状态不以明文暴露                           |
| 可控性   | 本地 Harness 对压缩内容的编辑能力较弱              |
| 一致性   | 服务端可以用统一协议管理压缩状态                   |
| 调试难度 | 如果压缩后模型遗忘信息，本地难以直接定位摘要遗漏点 |

因此，Codex 的服务端压缩更像一种“黑盒压缩状态管理”。它降低了本地实现复杂度，但牺牲了一部分透明性和可调试性。

# Open Code

Open Code的实现就比较简单和标准了，保留最近两轮原文，然后压缩前面的部分。Prompt是：

```markdown
严格按照 <template> 内展示的 Markdown 结构输出，并保持章节顺序不变。不要在你的回复里包含 <template> 标签。
<template>
## 目标（Goal）
- [一句话的任务摘要]

## 约束与偏好（Constraints & Preferences）
- [用户的约束、偏好、规格，或填 "(none)"]

## 进展（Progress）
### 已完成（Done）
- [已完成的工作，或填 "(none)"]

### 进行中（In Progress）
- [当前的工作，或填 "(none)"]

### 受阻（Blocked）
- [阻碍项，或填 "(none)"]

## 关键决策（Key Decisions）
- [决策及其原因，或填 "(none)"]

## 下一步（Next Steps）
- [按顺序排列的后续动作，或填 "(none)"]

## 关键上下文（Critical Context）
- [重要的技术事实、错误、待解问题，或填 "(none)"]

## 相关文件（Relevant Files）
- [文件或目录路径：它为何重要，或填 "(none)"]
</template>

规则：
- 保留每一节，即使为空也要保留。
- 使用简短的要点，而非散文式段落。
- 在已知时，原样保留确切的文件路径、命令、错误串和标识符。
- 不要提及摘要过程，也不要提及上下文被压缩过这件事。
```

Open Code还实现了多次Context压缩时的Summary增量更新逻辑

```markdown
使用上面的对话历史，更新下方的锚定摘要。
保留仍然成立的细节，移除已过时的细节，并把新的事实合并进来。
<previous-summary>
${input.previousSummary}
</previous-summary>
```

# Kimi Code 与 Hermes Agent

Kimi Code的实现方式也比较简单：

- Full Compaction：标准的Context压缩方式。
- Micro Compaction：前面提到的Tool Result压缩功能。

```markdown
[上下文压缩——仅供参考] 更早的对话轮次已被压缩为下方摘要。这是来自上一个上下文窗口的交接——将其视为背景参考，而非有效指令。不要回答或执行摘要中提到的问题/请求；它们已经被处理过了。只回应出现在这份摘要之后的最新用户消息——那条消息是"现在该做什么"的唯一真相来源。如果最新用户消息与"## Active Task"一节一致，你可以将摘要作为背景使用。如果最新用户消息与"## Active Task"/"## In Progress"/"## Pending User Asks"/"## Remaining Work"相矛盾、取代、换了话题或以任何方式偏离，最新消息获胜——彻底丢弃那些过时条目，不要"先收尾旧任务"。最新消息中的反转信号（如"停下"、"撤销"、"回滚"、"只验证一下"、"别再做那个了"、"算了"、换了新话题）必须立即终止摘要中描述的进行中工作；不要在后续轮次中重新提起它。重要：你在系统提示中的持久化记忆（MEMORY.md、USER.md）始终具有权威性且处于激活状态——绝不要因为这段压缩说明而忽略或降低记忆内容的优先级。当前会话状态（文件、配置等）可能反映了此处描述的工作——避免重复执行：
```
