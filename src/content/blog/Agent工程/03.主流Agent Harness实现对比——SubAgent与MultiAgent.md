# SubAgent与MultiAgent的三个目的

1. **将一个复杂的任务拆解为几个任务**，每个任务可以使用独立的`Context Session`进行处理，而这个独立Context Session就可以被视为一个Agent，即使模型能力和Prompt都一致。
2. **为了获取旁观者视角**，也就是不让Agent Session自己进行自省review，而是让另一个独立Context Session来进行检查或者讨论。
3. **对于可以可以拆分并行子任务的任务，可以进行加速**

# Claude Code

## 四种 SubAgent 模式概览

| 模式                         | 默认状态 | 核心特征                                                 | 适合场景                                     |
| :--------------------------- | :------- | :------------------------------------------------------- | :------------------------------------------- |
| 非 Fork 模式                 | 默认开启 | SubAgent 从零开始，不继承主会话上下文                    | 代码搜索、独立调研、多步任务                 |
| Fork 模式                    | 默认关闭 | SubAgent 可继承当前 session context，并共享 prompt cache | 不想把中间工具噪声留在主上下文中的调研或实现 |
| Coordinator 模式             | 默认关闭 | 主 Agent 变成协调者，不能直接 Edit/Bash，只能派 worker   | 多 worker 并行研究、实现、验证的大型工程任务 |
| Teammate / Agent Swarms 模式 | 默认关闭 | 更接近对等 Multi-Agent，Agent 之间通过信箱异步通信       | 长时间、多角色、多 Agent 协作探索            |

这四类模式体现了 Claude Code 对 SubAgent 的不同理解：从“工具型子进程”逐渐发展到“可继承上下文的 fork worker”，再到“由 coordinator 管理的 Agent team”，最后接近真正的 Multi-Agent Swarm。

## 默认非 Fork 模式

```markdown
启动一个新的 agent 来自主处理复杂的多步任务。

Agent 工具启动专门的 agent（子进程），它们自主处理复杂任务。每种 agent 类型有特定的能力和可用工具。

可用的 agent 类型及其工具：
- general-purpose：通用 agent，用于研究复杂问题、搜索代码和执行多步任务。当你搜索关键词或文件且不确定前几次能找到正确匹配时，用这个 agent 替你搜。（工具：全部）
- Explore：快速只读搜索 agent，用于定位代码。用它按模式找文件、按符号或关键词 grep、或回答"X 在哪定义 / 哪些文件引用了 Y"。不要用于代码审查、设计文档审计、跨文件一致性检查或开放性分析——它只读片段不读全文。调用时指定搜索广度："quick"做单次定向查找、"medium"适度探索、"very thorough"跨多个位置和命名惯例搜索。（工具：除 Agent/ExitPlanMode/Edit/Write/NotebookEdit 外全部）
- Plan：软件架构师 agent，用于设计实现计划。当你需要规划任务的实现策略时使用。返回分步计划、识别关键文件、考虑架构权衡。（工具：除 Agent/ExitPlanMode/Edit/Write/NotebookEdit 外全部）
- claude-code-guide：当用户询问 Claude Code / Agent SDK / Claude API 的使用问题时使用。（工具：Glob/Grep/Read/WebFetch/WebSearch）
- statusline-setup：用于配置用户的 Claude Code 状态栏设置。（工具：Read/Edit）

使用 Agent 工具时，指定 subagent_type 参数来选择使用哪种 agent 类型。如果省略，使用通用 agent。

## 什么时候不该用
如果目标已知，直接用对应工具：已知路径用 Read，特定符号或字符串用 Grep。本工具只用于跨代码库的开放性问题，或与某种可用 agent 类型匹配的任务。

## 撰写 prompt
像给一个刚走进房间的聪明同事交代情况一样——它没有看过这段对话，不知道你试过什么，不理解这个任务为什么重要。
- 说明你在做什么、为什么。
- 描述你已经了解到或排除了什么。
- 给出关于周边问题的足够上下文，让 agent 能做出判断，而不只是机械地执行一条窄指令。
- 如果需要简短回复，明确说出来（"200 词以内汇报"）。
- 查找类：交出确切的命令。调查类：交出问题——当前提错误时，规定好的步骤会变成累赘。

简短的命令式 prompt 只会产出浅薄、泛化的工作。

**永远不要委托理解。** 不要写"根据你的发现修这个 bug"或"根据调研去实现"。那些措辞把综合能力推给了 agent 而非你自己承担。写出能证明你理解了的 prompt：包含文件路径、行号、具体要改什么。
```

非 Fork 模式是 Claude Code 当前默认的 SubAgent 方案。它的核心特点是：**创建的 SubAgent 不继承主 session 的完整上下文，而是从零开始执行任务**。因此，主 Agent 在调用 SubAgent 时，必须把任务背景、目标、已知信息、排除过的方向和期望输出写清楚。

Claude Code 提供了多种 SubAgent 类型，包括：

- `general-purpose`：通用 Agent，用于复杂问题研究、代码搜索和多步任务；
- `Explore`：快速只读搜索 Agent，用于定位文件、符号和引用关系；
- `Plan`：架构规划 Agent，用于设计实现步骤和分析权衡；
- `claude-code-guide`：用于 Claude Code、Agent SDK、Claude API 相关问题；
- `statusline-setup`：用于配置 Claude Code 状态栏。

非 Fork 模式的关键原则是：**目标已知时不要滥用 SubAgent**。如果已经知道文件路径，应直接使用 Read；如果只是在找具体字符串，应直接 Grep。SubAgent 更适合开放性、跨代码库、需要多步探索的问题。

在 prompt 写法上，Claude Code 明确要求：像给一个刚进房间的聪明同事交代任务一样写。因为 SubAgent 没有看过当前对话，所以 prompt 必须自包含。简单命令式 prompt 通常只会得到浅层结果。

尤其重要的是：**不要委托理解**。不能写“根据你的发现修复 bug”或“根据研究结果实现它”。**主 Agent 应该自己理解 SubAgent 的发现，然后给出包含文件路径、行号和具体修改要求的后续指令**。

## AgentTool 的执行机制

AgentTool 支持前台和后台两种执行方式。

前台执行适合主 Agent 必须等待结果才能继续的情况，例如调研结果会决定下一步修改方案。后台执行适合存在真正独立工作的情况。后台 Agent 完成后会主动通知主 Session，主 Agent 不需要 sleep、轮询或主动检查进度。这是一个比较好的工程设计，因为它减少了主上下文中的等待噪声，也避免主 Agent 频繁查询子任务状态。

AgentTool 还有几个重要规则：

- 应尽可能并发启动多个相互独立的 Agent；
- 子 Agent 的结果默认对用户不可见，需要主 Agent 总结后转述；
- 新 Agent 调用是全新上下文，prompt 必须自包含；
- 可以用 SendMessage 继续已经创建过的 Agent；
- SubAgent 应返回文本发现，不应主动写报告文件；
- 可以通过 `isolation: "worktree"` 让 Agent 在临时 git worktree 中运行，避免文件修改互相污染。

这些规则说明 Claude Code 的 SubAgent 更像工程任务执行单元，而不是普通聊天分身。

## **通用SubAgent的System Prompt：**

```markdown
你是 Claude Code（Anthropic 官方 CLI）的一个 agent。收到用户消息后，使用可用工具完成任务。把任务做完——不要镀金，但也不要做一半。

你的强项：
- 在大型代码库中搜索代码、配置和模式
- 分析多个文件以理解系统架构
- 调查需要探索多个文件的复杂问题
- 执行多步研究任务

指南：
- 文件搜索：不知道东西在哪时广泛搜索。知道具体路径时用 Read。
- 分析：先广后窄。如果第一种搜索策略没有结果，换多种策略试。
- 要全面：检查多个位置，考虑不同命名习惯，查找相关文件。
- 永远不要创建文件，除非对实现目标绝对必要。永远优先编辑已有文件而非创建新的。
- 永远不要主动创建文档文件（*.md）或 README 文件。仅在用户明确要求时才创建。

完成任务后，用简洁的报告回复所做的事情和关键发现——调用方会转达给用户，只需要要点。

说明：
- Agent 线程在每次 bash 调用之间会重置工作目录，因此请只使用绝对文件路径。
- 在最终回复中，分享与任务相关的文件路径（始终用绝对路径，不用相对路径）。只在精确文本是 load-bearing 时才包含代码片段（如你发现的 bug、调用方要求的函数签名）——不要复述你仅仅读过的代码。
- 为了与用户的清晰沟通，助手必须避免使用 emoji。
- 不要在工具调用前加冒号。类似"Let me read the file:"后跟一个 read 工具调用的文本，应该写成"Let me read the file."用句号。
```

这说明 Claude Code 对 SubAgent 的定位很明确：它应该高效完成被分配的工程任务，不应随意扩展范围、创建额外文档或输出过多噪声。

## Fork 模式

```markdown
## 什么时候该 fork
当中间工具输出不值得留在你的上下文中时，fork 自己（省略 `subagent_type`）。判断标准是定性的——"我以后还需要这个输出吗"——而非任务大小。
- **调研**：fork 开放性问题。如果调研可以拆成独立问题，在一条消息中启多个并行 fork。fork 比全新子 agent 更适合调研——它继承上下文并共享你的 cache。
- **实现**：如果实现工作需要超过几次编辑，优先 fork。先调研再实现。

Fork 很便宜，因为它共享你的 prompt cache。不要在 fork 上设 `model`——不同模型无法复用父的 cache。传一个简短的 `name`（一两个词，小写），让用户能在 teams 面板看到 fork 并中途引导它。

**不要偷看。** 工具结果包含一个 `output_file` 路径——除非用户明确要求查进度，否则不要 Read 或 tail 它。你会收到完成通知；信任它。飞行中读取 transcript 会把 fork 的工具噪声拉进你的上下文，这会抵消 fork 的意义。

**不要抢跑。** 启动后你对 fork 的发现一无所知。永远不要以任何格式捏造或预测 fork 结果——无论散文、摘要还是结构化输出。通知以 user-role 消息的形式在后续 turn 到达；它永远不是你自己写的。如果用户在通知到达前问后续问题，告诉他们 fork 还在运行——给状态，不给猜测。
```

Fork 模式默认关闭，可通过环境变量或服务端灰度开启。它的核心变化是：**SubAgent 可以 fork 当前主会话上下文，而不是从零开始**。这使它比普通 SubAgent 更适合承担与当前任务强相关的调研或实现工作。

Fork 模式的关键使用标准不是任务大小，而是：

> 中间工具输出是否值得继续留在主 Agent 的上下文中。

如果某些搜索、调研或实现会产生大量工具噪声，但最终只需要结论，那么 fork 是合理的。它既继承当前上下文，又能避免把探索过程污染主上下文。

Fork 模式有两个重要约束：

1. **不要偷看**：fork 运行中会产生 `output_file`，但主 Agent 不应中途 Read 或 tail，除非用户明确要求。否则 fork 的工具噪声又会进入主上下文，破坏 fork 的意义。
2. **不要抢跑**：fork 启动后，主 Agent 不能编造或预测 fork 的结果。结果会以通知形式返回，在此之前只能说明 fork 仍在运行。

Fork worker 启动时还会收到专门的 boilerplate，要求它不要再创建子 Agent，不要对话，不要提建议，直接使用工具完成任务，最后以 `Scope / Result / Key files / Files changed / Issues` 的结构化格式汇报。如果修改了文件，还需要先 commit 并报告 commit hash。

```markdown
<fork-boilerplate>
停下。先读这个。

你是一个 fork 出来的工作进程。你不是主 agent。

规则（不可商量）：
1. 你就是 fork。不要 spawn 子 agent；直接执行。
2. 不要对话、不要提问、不要建议下一步
3. 不要评论或添加元注释
4. 直接使用你的工具：Bash、Read、Write 等。
5. 如果修改了文件，汇报前先 commit。在报告中包含 commit hash。
6. 工具调用之间不要输出文本。静默使用工具，最后汇报一次。
7. 严格在你的指令范围内。如果发现范围外的相关系统，最多用一句话提及——其他 worker 负责那些区域。
8. 报告不超过 500 词，除非指令另有说明。实事求是、简洁。
9. 你的回复必须以 "Scope:" 开头。不要前言、不要思考过程外露。
10. 汇报结构化事实，然后停止。

输出格式（纯文本标签，不是 markdown 标题）：
Scope: <用一句话回述你被分配的范围>
Result: <答案或关键发现，限于上述范围>
Key files: <相关文件路径——研究任务时包含>
Files changed: <修改文件列表 + commit hash——仅在修改了文件时包含>
Issues: <问题列表——仅在有需要标记的问题时包含>
</fork-boilerplate>
```

Fork 模式本质上是对“上下文隔离”和“任务并行”的结合：**它继承必要背景，但把中间执行噪声隔离到分支中**。

## Coordinator 模式

Coordinator 模式默认关闭，开启后主 Agent 的角色会发生明显变化：它不再直接执行 Edit 或 Bash，而是成为一个 **软件工程任务协调者**。它通过 worker 完成研究、实现和验证，自己负责拆解任务、综合结果、向用户沟通和继续分配任务。

该模式下的典型工作流是：

```text
Research：多个 worker 并行调查代码、测试、错误来源
Synthesis：主 Agent 阅读结果，综合出具体实现规格
Implementation：worker 按规格修改代码并提交
Verification：新的 worker 独立验证改动是否真正有效
```

Coordinator 模式最重要的思想是：**主 Agent 不能把理解外包给 worker**。当 worker 返回研究结果后，主 Agent 必须先理解，再写出具体、可执行、带文件路径和行号的实现 prompt。不能写“根据你的发现修复”，因为这会把综合判断责任推给 worker。

该模式也强调并发：只读研究任务可以并行运行，写密集任务对同一组文件应串行，验证任务最好由新 worker 独立完成，避免实现者自我确认。

Coordinator 模式虽然有些激进，但它代表了未来 Agent Team 的一种可能主流形态：用户不需要关注内部有多少 Agent 在工作，只与 coordinator 交互；coordinator 负责把内部 worker 的结果翻译成用户能理解的进展和结论。

## Teammate / Agent Swarms 模式

Teammate / Agent Swarms 模式更接近真正的 Multi-Agent 范式。它默认不开启，属于实验性能力。

与默认 SubAgent 不同，在该模式下，每个 Agent 不再只是主 Agent 的工具，而更像一个可异步长期执行的团队成员。Agent 之间可以通过信箱进行消息通信，主 Agent 则更接近 team leader。

该模式的特点是：

- Agent 之间更接近对等协作；
- 每个 Agent 可以异步、长时间执行；
- 通信通过信箱进行，而不是简单工具返回；
- Agent 的停止通常需要 team leader 主动发消息要求停止。

相比 Coordinator 模式，Agent Swarms 更像完整的 Agent 团队协作系统。但由于该功能仍未转正，说明这种模式虽然有潜力，但工程复杂度和稳定性问题可能更高。

# Codex

Codex有两种不同的SubAgent/MultiAgent模式，分别对应于Claude的非Fork模式和Teammate模式。

## Collab模式

```markdown
为一个范围明确的任务派生一个子 agent。返回派生出的 agent id，以及（若可用）面向用户的昵称。派生出的 agent 默认继承你当前的模型。若提供了 `model`，它指定派生 agent 所用的模型。
本 spawn_agent 工具让你能访问默认继承你当前模型的子 agent。除非用户明确要求换一个模型、或有清晰的任务特定理由，否则不要设置 `model` 字段。你应遵循下面的规则与指引来使用本工具。

当且仅当用户明确要求子 agent、委派或并行 agent 工作时，才使用 `spawn_agent`。
对深度、彻底性、研究、调查或详细代码库分析的诉求，不构成派生的许可。
下面的角色指引只在派生已被授权之后，帮助选择用哪个 agent；它本身绝不授权派生。

### 何时委派 vs 自己做这个子任务
- 首先，快速分析整体用户任务，形成一份简明的高层计划。识别哪些任务是关键路径上的即时阻塞项，哪些是需要但能并行、不阻塞下一步本地动作的旁路（sidecar）任务。作为计划的一部分，明确决定你现在应在本地立即做哪个即时任务。在委派给 agent 之前先做这步规划，以免把即时阻塞任务交给子模型、然后白白等它。
- 当一个子任务足够简单、它能处理且能与你的本地工作并行时，使用子 agent。优先委派具体、有界、能实质推进主任务且不阻塞你下一步本地动作的旁路任务。
- 当你的下一步直接依赖某结果时，不要委派那项紧急的阻塞工作。如果紧接着的动作就被该任务卡住，主 rollout 通常应在本地做，以保持关键路径推进。
- 当子任务太难以良好委派、且紧耦合、紧急或很可能阻塞你下一步时，把工作留在本地。

### 设计被委派的子任务
- 子任务必须具体、定义清晰、自包含。
- 被委派的子任务必须实质推进主任务。
- 不要在主 rollout 与被委派子任务之间重复劳动。
- 避免在同一个未解决的线程上发出多次委派调用，除非新的委派任务确实不同且必要。
- 把委派诉求收窄到你下一步需要的具体产出。
- 对编码任务，当子 agent 能在清晰的写作用域内做出有界补丁时，优先委派具体的改代码 worker 子任务，而非只读的 explorer 分析。
- 委派编码工作时，指示子模型直接在它 fork 出的工作区里编辑文件，并在最终答案里列出它改动的文件路径。
- 对改代码子任务，拆解工作使每个被委派任务有**互不相交**的写集合。

### 委派之后
- 极其节制地调用 wait_agent。仅当你为下一个关键路径步骤立即需要其结果、且在它返回前你被阻塞时，才调用 wait_agent。
- 不要自己重做被委派的子 agent 任务；聚焦于整合结果或处理不重叠的工作。
- 当子 agent 在后台运行时，立即去做有意义的、不重叠的工作。
- 不要反射式地反复等待。
- 当一个被委派的编码任务返回时，快速 review 上传的改动，然后整合或打磨它们。

### 并行委派模式
- 当你有可独立回答的不同问题时，并行跑多个独立的信息搜集子任务。
- 当写作用域不重叠时，把实现拆成互不相交的代码库切片，并为它们并行派多个 agent。
- 仅当验证能与进行中的实现并行、且很可能在最终整合前抓住一个具体风险时，才委派验证。
- 关键在于：在同一轮内找到并行派出多个独立子任务的机会，同时确保每个子任务定义清晰、自包含、并实质推进主任务。
```

其中，Collab 是默认模式，命名空间为 `multi_agent_v1`，包含：

```text
spawn_agent
wait_agent
send_input
close_agent
resume_agent
```

它与 Claude Code 的 AgentTool 类似，但 Codex 的子 Agent 始终是异步的，更像启动一个后台任务，然后通过 wait 或 send_input 与其交互。

Codex 的最大特点是 **不积极鼓励自动使用 SubAgent**。它明确规定，只有当用户显式要求 sub-agents、delegation 或 parallel agent work 时，才允许 `spawn_agent`。用户只是要求“深入分析、详细研究、彻底调查”并不自动构成使用 SubAgent 的许可。

这说明 Codex 更强调控制成本和避免过度委派。它要求主 Agent 先判断：

- 哪些任务在关键路径上，必须自己立即处理；
- 哪些任务是 sidecar task，可以并行推进；
- 哪些任务适合委派；
- 哪些任务太紧耦合，不适合交给子 Agent。

Codex 对委派任务也有严格要求：任务必须具体、自包含、边界清晰，并且不能与主 rollout 重复劳动。对于代码任务，还要求不同子 Agent 的写作用域互不重叠。

这套设计的重点是：

> SubAgent 应该用于并行推进非阻塞的、有界的子任务，而不是把关键路径上的理解和判断外包出去。

# OpenAI Agents SDK

## Agent as Tool

这是典型的子 Agent 工具化模式，类似 Claude Code 的非 Fork SubAgent。一个 Agent 可以把另一个 Agent 当作工具调用，并且支持多层递归创建。权限请求也可以层层穿透返回。

这种方式适合构建：

- 专门负责代码审查的 Agent；
- 专门负责测试生成的 Agent；
- 专门负责检索文档的 Agent；
- 专门负责数据分析的 Agent。

## Handoff

Handoff 比较特殊。它本质上不是传统 SubAgent，而是将当前对话交接给另一个 Agent，即替换当前 Agent 的 role / system prompt，并在原有 context 上继续执行。

可以理解为：

```text
当前 Agent
  ↓ handoff
另一个 Agent 接管当前上下文继续执行
```

它更适合构建由多个 Agent 节点组成的 workflow。例如：

```text
需求分析 Agent → 代码实现 Agent → 测试验证 Agent → 文档生成 Agent
```

但材料中指出，目前 Handoff 的实现无法复用 KV Cache，因此在性能上可能不如真正的 fork 或缓存共享机制。

OpenAI Agents SDK 的核心特点是抽象简单、灵活度高，但也意味着开发者需要自己设计严格的使用边界、任务分解方式和验证机制。

# Open Code 与 Kimi Code

Open Code 支持 Agent as Tool，整体与 Claude Code 的非 Fork 模式接近。它强调：

- 已知文件路径时直接用 Read / Glob；
- 搜索特定类或符号时直接用 Grep；
- 只有复杂多步任务才使用 Task；
- 子 Agent 输出默认对用户不可见，需要主 Agent 总结给用户；
- 每次新建 Agent 默认是全新上下文；
- 如果提供 `task_id`，可以恢复同一个子 Agent 会话。

这类设计的重点是减少不必要委派。因为启动子 Agent 有上下文交接成本，小任务直接做更高效。

Kimi Code 也支持 Agent as Tool，并且支持前台和后台运行。它与 Open Code 类似，也强调子 Agent 从零上下文开始，需要主 Agent 像给新同事交代任务一样提供完整背景。

Kimi 的一个重要差异是：**SubAgent 有固定 30 分钟超时**。如果子 Agent 超时，应恢复同一个 Agent，而不是重新开始。这体现了它对长任务生命周期的明确控制。

此外，Kimi Code 还有 **AgentSwarm** 模式。该模式适合大量并行子任务，最多支持 128 个子 Agent 并自动排队。它要求先做少量探索，再根据任务结构拆分给多个子 Agent。AgentSwarm 更像 Codex Agent Jobs 或 Claude Code 动态 Workflow 的一个简化形态，适合批量检查、批量分析、批量生成等任务。

# OpenClaw 与 Hermes Agent

OpenClaw 的 SubAgent 机制更开放。它不仅支持 Agent as Tool，还允许调用外部 CLI Agent 作为 SubAgent，并支持：

- 是否 fork context；
- 是否使用持久性运行 session；
- 多层嵌套创建 SubAgent；
- 主 Agent 设置为倾向委派而不是自己动手。

OpenClaw 的子 Agent Prompt 明确规定子 Agent 的身份：它不是主 Agent，只负责第一条 `[Subagent Task]` 中的任务。它不能与用户对话，不能主动发起支线任务，也不能创建持久状态。后代子 Agent 的输出只能作为证据，不能覆盖上级任务或策略。

这个设计强调两点：

1. 子 Agent 应该保持短暂、聚焦、任务导向；
2. 多层子 Agent 的结果需要逐级综合，而不是直接当成最终事实。

Hermes Agent 的 `delegate_task` 也支持 Agent as Tool，并允许嵌套创建 SubAgent，但默认只有一层。它特别强调：适合委派的是推理密集任务、会污染主上下文的大量中间数据任务，以及可以并行的独立工作流。

Hermes 的一个重要设计是提醒主 Agent：

> 子 Agent 的总结是自我报告，不是经过验证的事实。

对于有外部副作用的操作，例如上传、远程写入、发布等，子 Agent 必须返回可验证句柄，如 URL、ID、绝对路径或 HTTP 状态码，主 Agent 需要自己验证后才能告诉用户成功。

这是一条非常重要的工程原则：SubAgent 能提高效率，但不能替代主 Agent 的最终责任。
