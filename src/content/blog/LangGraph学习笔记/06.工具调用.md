# 06. 工具调用

## 复习抓手

- 工具调用链路：定义工具 -> 绑定到模型 -> 模型产生 tool_calls -> 工具节点执行 -> 结果回写 State。
- 工具描述要明确，让模型能判断何时调用、如何传参。
- 工具执行结果需要回到消息列表或 State，供下一轮模型推理使用。
- 结合条件循环后，就形成常见的 ReAct/Tool-Calling Agent 图。

## 工具定义、绑定、调用和执行

## 工具的定义

```python
from langchain_core.messages import AIMessage
from langchain_core.tools import tool
from langgraph.prebuilt import ToolNode

@tool
def get_weather(location: str):
    """调用此函数获取当前天气。"""
    if location.lower() in ["北京", "深圳"]:
        return "现在是20度，有雾。"
    else:
        return "现在是10度，晴朗。"

@tool
def get_coolest_cities():
    """获取最冷城市列表"""
    return "哈尔滨, 北京"

tools = [get_weather, get_coolest_cities]
tool_node = ToolNode(tools)
```

langgraph提供了低层面的封装，可以手动执行工具

```python
message_with_single_tool_call = AIMessage(
    content="",
    tool_calls=[
        {
            "name": "get_weather",
            "args": {"location": "北京"},
            "id": "tool_call_id",
            "type": "tool_call",
        },
    ],
)

tool_node.invoke({"messages": [message_with_single_tool_call]})
```

## 复习补充

工具调用让 LangGraph 从“只会生成文本”变成“能执行动作”。在图里，工具调用通常不是单点功能，而是一条闭环：模型决定调用工具，工具节点执行，结果写回 State，模型再根据结果继续推理。

| 步骤 | 关键点 | 易错点 |
| --- | --- | --- |
| 定义工具 | 名称、描述、参数清晰 | 描述含糊会导致模型误用 |
| 绑定工具 | 让模型知道可用工具 | 模型不支持工具调用时会失败 |
| 解析 tool_calls | 读取模型想调用什么 | 参数结构要和工具 schema 对齐 |
| 执行工具 | 真正调用外部能力 | 要处理异常、超时、权限 |
| 回写结果 | 把工具结果交还模型 | 不回写就无法继续推理 |

### 工程提醒

工具描述要写给模型看，不是写给人看。一个好的工具描述应该说明什么时候用、输入参数是什么、返回结果代表什么。生产环境中还要考虑权限、审计、限流、重试和结果脱敏。

### 复习检查清单

- 能否画出模型节点和工具节点之间的循环。
- 能否解释 `bind_tools` 的作用。
- 能否判断某个函数是否适合暴露为工具。
- 能否说明工具执行结果为什么要回写到消息或 State。
