# 索引优化

# **句子窗口检索**

检索时聚焦于高度精确的单个句子，在送入LLM生成答案前，又智能地将上下文扩展回一个更宽的“窗口”，从而同时保证检索的准确性和生成的质量。

| 阶段 | 做什么 | 目的 |
| --- | --- | --- |
| 索引阶段 | 把文档切成单个句子，每个句子作为一个 Node（节点）存入向量数据库 | 保证检索粒度足够细 |
| 元数据保存 | 每个句子节点保存前后 N 个句子组成的 window（窗口） | 提前准备上下文 |
| 检索阶段 | 对单个句子节点做相似度搜索 | 精确找到相关核心句子 |
| 后处理阶段 | 用 MetadataReplacementPostProcessor（元数据替换后处理器）把句子替换成上下文窗口 | 扩展上下文 |
| 生成阶段 | 把扩展后的上下文交给 LLM（大语言模型）回答 | 提高答案完整性和连贯性 |

# 结构化索引

在索引文本块的同时，为其附加结构化的**元数据（Metadata）**。这些元数据可以是任何有助于筛选和定位信息的标签

通过这种方式，可以在检索时实现“元数据过滤”和“向量搜索”的结合。例如，当用户查询“请总结一下2023年第二季度财报中关于AI的论述”时，系统可以：

（1）**元数据预过滤**：首先通过元数据筛选，只在 `document_type == '财报'`、`year == 2023` 且 `quarter == 'Q2'` 的文档子集中进行搜索。

（2）**向量搜索**：然后，在经过滤的、范围更小的文本块集合中，执行针对查询“关于AI的论述”的向量相似度搜索。

这种“**先过滤，再搜索**”的策略，能够极大地缩小检索范围，显著提升大规模知识库场景下RAG应用的检索效率和准确性。LlamaIndex 提供了包括“自动检索”（Auto-Retrieval）在内的多种工具来支持这种结构化的检索范式。

```python
原始文档
  ↓
文本分块
  ↓
为每个文本块附加元数据
  ↓
建立向量索引
  ↓
用户查询
  ↓
先用元数据过滤范围
  ↓
再在小范围内做向量相似度检索
  ↓
返回更相关的上下文
```

# 递归检索

页面中的示例流程如下：

| 步骤 | 作用 |
| --- | --- |
| 为每个 Sheet（工作表）创建 PandasQueryEngine（Pandas 查询引擎） | 每个表格有自己的查询引擎 |
| 为每个 Sheet（工作表）创建 IndexNode（索引节点） | 每个节点是一段摘要，相当于指向表格的“指针” |
| 用所有 IndexNode（索引节点）建立顶层向量索引 | 顶层索引用来判断该进入哪个表 |
| 创建 RecursiveRetriever（递归检索器） | 负责从顶层节点跳转到对应查询引擎 |
| 执行用户查询 | 系统先选中年份对应的表，再在表内查询 |

```python
用户问题
  ↓
顶层索引：判断问题属于哪个数据源
  ↓
进入对应子索引 / 子查询引擎
  ↓
在局部范围内精确查询
  ↓
返回答案
```

# 路由和检索彻底分离

| 阶段 | 做法 |
| --- | --- |
| 第一步：路由 | 为每个工作表创建摘要索引，例如“此文档包含 1994 年电影信息” |
| 第二步：过滤检索 | 根据路由结果获得目标 `sheet_name`，再在内容索引中使用 MetadataFilter（元数据过滤器）限制检索范围 |

```python
用户问题
  ↓
摘要索引判断目标数据源
  ↓
得到 sheet_name == "年份_1994"
  ↓
内容索引中使用元数据过滤
  ↓
只在 1994 年工作表内容中检索
  ↓
生成答案
```
