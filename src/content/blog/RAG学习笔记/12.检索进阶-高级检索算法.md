# 检索进阶-高级检索算法

解决 **RAG（检索增强生成）系统中“检索结果不够准、上下文太长、检索失败导致幻觉”** 这三个问题。文章主要讲了三类高级技术：**重排序 Re-ranking（重排序）**、**压缩 Compression（压缩）**、**校正 Correcting（校正）**。

```python
用户问题
   ↓
基础检索：向量检索 / 混合检索
   ↓
重排序：把真正相关的文档排到前面
   ↓
压缩：删除无关文本，只保留关键内容
   ↓
校正：判断检索结果是否可靠
   ↓
LLM（大语言模型）生成答案
```

# 重排序 Re-ranking（重排序）

在初步召回的候选文档中，再用更精细的方法重新计算相关性，让真正有用的文档排到前面。

```python
用户问题
   ↓
基础检索召回 Top-50
   ↓
重排序模型重新打分
   ↓
选择 Top-5 或 Top-10
   ↓
交给 LLM 生成答案
```

## 基于大语言模型的重排序

既然最终是 LLM（大语言模型）根据上下文回答问题，那可以直接让 LLM 判断哪些文档最相关。

```python
用户问题 + 候选文档列表
        ↓
LLM 阅读并判断相关性
        ↓
输出文档排序和相关性分数
        ↓
选取高分文档作为最终上下文
```

| 场景 | 原因 |
| --- | --- |
| 高价值问答 | 可以接受较高成本，换取更好的语义判断 |
| 文档数量不多 | LLM 输入窗口有限，不适合放太多候选文档 |
| 需要强语义理解 | LLM 比普通向量相似度更擅长理解复杂意图 |

## Cross-Encoder（交叉编码器）重排

Cross-Encoder（交叉编码器）是一种高精度重排序方法。它会把 query（查询）和每个 document（文档）拼接成一个整体输入，输出一个相关性分数。文章指出，它的高精度来自于同时分析查询和文档，但高延迟也来自于每个“查询-文档”对都要单独推理一次。

```python
基础检索召回 Top-50
        ↓
Query + Doc1 → Cross-Encoder → score1
Query + Doc2 → Cross-Encoder → score2
...
Query + Doc50 → Cross-Encoder → score50
        ↓
按 score 重新排序
        ↓
返回最终 Top-K
```

| 场景 | 原因 |
| --- | --- |
| Top-K 精排 | 候选数量已经较少，可以承受逐个打分 |
| 高准确率要求 | 比单纯向量相似度更能判断细粒度相关性 |
| 问答系统后处理 | 适合作为召回后的第二阶段排序 |

## ColBERT（基于 BERT 的上下文化后期交互模型）重排

分别对查询和文档中的每个 token（词元）进行编码，然后在查询阶段计算 token（词元）级别的最大相似度，也就是 MaxSim（最大相似度）。

```python
文档端：
Document → Token Embeddings（词元向量） → 预先存储

查询端：
Query → Token Embeddings（词元向量）

匹配阶段：
每个查询 token 与文档 token 计算最大相似度
        ↓
所有 token 最大相似度求和
        ↓
得到最终相关性分数
```

## 重拍方法对比

| 特性 | RRF | RankLLM | Cross-Encoder | ColBERT |
| --- | --- | --- | --- | --- |
| **核心机制** | 融合多个排名 | LLM 推理，生成排序列表 | 联合编码查询与文档，计算单一相关分 | 独立编码，后期交互 |
| **计算成本** | 低（简单数学计算） | 中 (API 费用与延迟) | 高（N次模型推理） | 中（向量点积计算） |
| **交互粒度** | 无（仅排名） | 概念/语义级 | 句子级（Query-Doc Pair） | Token 级 |
| **适用场景** | 多路召回结果融合 | 高价值语义理解场景 | Top-K 精排 | Top-K 重排 |

# 压缩

## LangChain 的上下文压缩检索器

| 压缩器 | 作用 | 类型 |
| --- | --- | --- |
| `LLMChainExtractor` | 用 LLM 从每篇文档中抽取相关片段 | 内容提取 |
| `LLMChainFilter` | 用 LLM 判断整篇文档是否相关 | 文档过滤 |
| `EmbeddingsFilter` | 用向量相似度过滤低相关文档 | 快速过滤 |

```python
        ↓
ColBERT 重排，选出更相关的 Top-5
        ↓
LLMChainExtractor 提取相关句子
        ↓
返回压缩后的最终上下文
```

# 校正 Correcting

检索出来的文档是相关且正确的。但现实中检索可能失败，返回不相关、过时甚至错误的文档。如果把这些错误上下文直接交给 LLM（大语言模型），就可能导致 hallucination（幻觉）。

不盲目信任检索结果，而是在生成答案之前先评估检索结果质量。

## C-RAG（校正型检索增强生成）的三阶段流程

```python
1. Retrieve（检索）
从内部知识库中召回相关文档

2. Assess（评估）
检索评估器判断文档质量：
Correct（正确）
Incorrect（不正确）
Ambiguous（模糊）

3. Act（行动）
根据评估结果采取不同策略
```

| 评估结果 | 系统动作 |
| --- | --- |
| Correct（正确） | 进入 Knowledge Refinement（知识精炼），把文档切成更小知识片段，过滤无关内容，再组合成上下文 |
| Incorrect（不正确） | 触发 Knowledge Searching（知识搜索），说明内部知识库不可靠，需要查询重写后进行 Web 搜索 |
| Ambiguous（模糊） | 也触发外部搜索，用额外信息辅助回答 |
