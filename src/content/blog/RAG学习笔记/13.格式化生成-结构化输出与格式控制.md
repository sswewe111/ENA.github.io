# 格式化生成 - 结构化输出与格式控制

**如何让大语言模型不只是输出一段自然语言，而是输出固定格式的数据**，例如 `JSON（结构化数据格式）`、`XML（可扩展标记语言）`、Python 对象等。这样模型输出的内容才能被程序直接读取、校验、存储或传给外部工具使用。

# 格式化生成的几种实现方式

| 方法 | 核心思想 | 优点 | 缺点 |
| --- | --- | --- | --- |
| LangChain Output Parsers（输出解析器） | 在提示词中注入格式要求，再解析模型输出 | 使用方便，适合工程开发 | 仍然依赖模型是否按要求输出 |
| LlamaIndex Structured Output（结构化输出） | 在 RAG 响应生成阶段直接约束输出结构 | 和 RAG 流程结合紧密 | 依赖框架能力 |
| 不依赖框架的提示工程 | 手写提示词，要求模型只返回 JSON | 简单、轻量 | 稳定性一般 |
| Function Calling（函数调用） | 模型返回工具调用参数，由代码执行工具 | 最可靠，适合 Agent 和真实系统 | 实现流程更复杂 |

## LangChain Output Parsers（输出解析器）

一边告诉模型要输出什么格式，一边把模型返回的字符串解析成程序对象

PydanticOutputParser的具体流程

| 步骤 | 作用 |
| --- | --- |
| 定义 Pydantic 模型 | 明确输出字段、字段类型、字段含义 |
| 创建解析器 | 根据模型生成格式化指令 |
| 构造 Prompt（提示词） | 把格式要求放进提示词 |
| 调用 LLM（大语言模型） | 让模型生成 JSON 字符串 |
| 解析和验证 | 将 JSON 转成 Python 对象，并检查类型是否正确 |

```python
# (此处省略了导入和 LLM 初始化代码)

# 1. 定义期望的数据结构
class PersonInfo(BaseModel):
    """用于存储个人信息的数据结构。"""
    name: str = Field(description="人物姓名")
    age: int = Field(description="人物年龄")
    skills: List[str] = Field(description="技能列表")

# 2. 基于 Pydantic 模型，创建解析器
parser = PydanticOutputParser(pydantic_object=PersonInfo)

# 3. 创建提示模板，注入格式指令
prompt = PromptTemplate(
    template="请根据以下文本提取信息。\n{format_instructions}\n{text}\n",
    input_variables=["text"],
    partial_variables={"format_instructions": parser.get_format_instructions()},
)

# 4. 创建处理链 (假定 llm 已被初始化)
chain = prompt | llm | parser

# 5. 执行调用
text = "张三今年30岁，他擅长Python和Go语言。"
result = chain.invoke({"text": text})

# 6. 打印结果
print(result)
# name='张三' age=30 skills=['Python', 'Go语言']
```

## LlamaIndex 的输出解析

LlamaIndex 的输出解析和 RAG 流程结合更紧密。它主要涉及两个概念：

| 概念 | 含义 |
| --- | --- |
| Response Synthesis（响应合成） | 检索到多个文本块后，如何组织它们并生成最终答案 |
| Structured Output（结构化输出） | 让最终答案不是普通文本，而是符合指定结构的数据 |

在 RAG 中，检索器先召回相关文本块，也就是 Nodes（节点），然后 Response Synthesizer（响应合成器）负责把这些文本块交给模型生成答案。常见方式包括 `refine（逐步优化）` 和 `compact（压缩整合）`。当需要结构化输出时，LlamaIndex 也可以用 Pydantic 模型约束输出.

## 不依赖框架的实现方式

| 技巧 | 说明 |
| --- | --- |
| 明确要求 JSON | 例如：“你必须只返回 JSON，不要输出解释文字” |
| 提供 JSON Schema | 告诉模型每个字段是什么类型、什么意思 |
| 提供 Few-shot（少样本示例） | 给 1 到 2 个输入和输出示例，让模型模仿 |
| 使用语法约束 | 对本地模型使用 GBNF 等方式强制限制生成格式 |

## Function Calling（函数调用）(重点)

函数调用的流程：

```python
用户问题
↓
模型判断是否需要调用工具
↓
模型返回 tool_calls（工具调用请求）
↓
代码解析工具名和参数
↓
代码真正执行工具/API
↓
工具结果返回给模型
↓
模型生成最终回答
```

| 步骤 | 作用 |
| --- | --- |
| 定义 tools（工具） | 用 JSON Schema 描述工具名称、功能、参数 |
| 第一次调用模型 | 让模型判断是否需要工具 |
| 处理 tool_calls | 从模型响应中解析工具名和参数 |
| 执行工具 | 由代码真正调用 API 或函数 |
| 第二次调用模型 | 把工具结果返回给模型 |
| 生成最终答案 | 模型基于真实工具结果回答用户 |

优势：

| 优势 | 解释 |
| --- | --- |
| 可靠性更高 | 它是模型原生支持的结构化调用能力，比单纯提示模型输出 JSON 更稳定 |
| 能做意图识别 | 模型不只是输出格式，还能判断用户想调用哪个工具 |
| 能连接外部世界 | 可以查数据库、调用 API、控制设备，是 Agent 的基础 |
