# RAG系统评估

一个 RAG 系统回答得好不好，不能只看最终答案顺不顺，而要拆开看：检索内容是否相关、生成答案是否忠实、最终回答是否真正回答了用户问题。

RAG 评估主要用于解决两个问题：**开发者如何量化追踪和优化 RAG 系统性能；用户或决策者如何客观比较不同 RAG 应用的优劣。**

# RAG 评估三元组

| 评估维度 | 中文含义 | 主要评估对象 | 核心问题 |
| --- | --- | --- | --- |
| Context Relevance（上下文相关性） | 检索内容是否和问题相关 | Retriever（检索器） | 检索出来的文档是否有用 |
| Faithfulness / Groundedness（忠实度 / 可信度） | 答案是否基于检索上下文 | Generator（生成器） | 模型有没有胡编、幻觉 |
| Answer Relevance（答案相关性） | 最终答案是否回答用户问题 | 端到端系统 | 答案是否切题、完整、有效 |

**忠实度**看“答案是不是从材料里来的”；**答案相关性**看“答案是不是用户真正想要的”。

# RAG 评估工作流

```python
RAG 系统评估
├── 检索评估
│   └── 评估上下文相关性
└── 响应评估
    ├── 评估忠实度
    └── 评估答案相关性
```

## 检索评估

检索器能不能把真正相关的文档找出来，并且排在靠前的位置。

**Context Precision（上下文精确率）**：检索出来的前 k 个文档中，有多少是真正相关的。

**Context Recall（上下文召回率）**：所有应该被找回的相关文档中，系统找回了多少

**F1-Score（F1 分数）**：F1 分数是精确率和召回率的综合指标。F1 = 2 × Precision × Recall / (Precision + Recall)

**MRR（平均倒数排名）**：第一个相关文档是否排得足够靠前。

**MAP（平均准确率均值）**：MAP 会先计算每个查询的 AP（Average Precision，平均准确率），再对所有查询的 AP 取平均

## 响应评估

（1）**忠实度 / 可信度**：衡量生成的答案在多大程度上可以由给定的上下文所证实。一个完全忠实的答案，其所有内容都必须能在上下文中找到依据，以此避免模型产生“幻觉”。

（2）**答案相关性**：衡量生成的答案与用户原始查询的对齐程度。一个高相关性的答案必须是直接的、切题的，并且不包含与问题无关的冗余信息。

| 方法 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- |
| 基于 LLM 的评估 | 能理解语义、逻辑、上下文 | 成本高，可能有评估偏见 | 忠实度、答案相关性、开放问答 |
| ROUGE / BLEU / METEOR | 计算快、成本低、客观稳定 | 不理解深层语义，容易误判同义表达 | 大规模初筛、有标准答案的数据集 |

| 指标 | 中文含义 | 主要关注点 |
| --- | --- | --- |
| ROUGE（面向召回的摘要评估指标） | 看标准答案中的内容被覆盖了多少 | 完整性 |
| BLEU（双语评估替补指标） | 看生成答案中的词有多少是有效的 | 准确性、流畅度 |
| METEOR（显式排序翻译评估指标） | 综合考虑精确率、召回率、词形和同义词 | 综合语义相似性 |

# 三个评估工具

| 工具 | 核心定位 | 更适合的阶段 |
| --- | --- | --- |
| LlamaIndex Evaluation | LlamaIndex 框架内置评估模块 | 开发阶段、调试阶段 |
| RAGAS | 独立的 RAG 评估框架 | 离线评估、策略对比、回归测试 |
| Phoenix | LLM 可观测性与评估平台 | 生产环境监控、Bad Case 分析 |

## LlamaIndex Evaluation

使用 LLM（大语言模型）作为“裁判”，自动对 RAG 系统的不同环节进行评分。

**很多情况下不需要提前准备人工标注的标准答案**，可以直接让 LLM 根据问题、上下文和答案进行判断。

工作流

| 步骤 | 作用 |
| --- | --- |
| 准备评估数据集 | 可以用 `DatasetGenerator` 自动从文档中生成问题-答案对，也可以加载已有数据集 |
| 构建查询引擎 | 搭建一个或多个待评估的 `QueryEngine` |
| 初始化评估器 | 选择 `FaithfulnessEvaluator`、`RelevancyEvaluator` 等评估器 |
| 执行批量评估 | 使用 `BatchEvalRunner` 对多个问题批量评估 |
| 分析结果 | 计算平均分，对比不同 RAG 策略优劣 |

LlamaIndex 支持的评估维度

| 指标 | 中文含义 | 评估对象 |
| --- | --- | --- |
| Faithfulness | 忠实度 | 答案是否基于检索上下文 |
| Relevancy | 相关性 | 答案是否切题 |
| Hit Rate | 命中率 | 检索内容是否包含正确答案 |
| MRR | 平均倒数排名 | 正确答案是否排得靠前 |

## RAGAS

RAGAS 的突出特点是支持无参考评估，在很多情况下不需要人工标注标准答案。

标准数据格式:

| 字段 | 含义 | 是否必须 |
| --- | --- | --- |
| question | 用户问题 | 必须 |
| answer | RAG 系统生成的答案 | 必须 |
| contexts | 检索到的上下文 | 必须 |
| ground_truth | 标准参考答案 | 部分指标需要 |

核心指标

| 指标 | 中文解释 | 主要回答的问题 |
| --- | --- | --- |
| faithfulness | 忠实度 | 答案中的信息是否能被上下文支持 |
| context_recall | 上下文召回率 | 标准答案所需信息是否被检索出来 |
| context_precision | 上下文精确率 | 检索上下文中有多少是真正有用的 |
| answer_relevancy | 答案相关性 | 答案是否切题 |

## Phoenix

在 RAG 系统中的定位不是单纯“打分”，而是更偏向 **生产环境中的追踪、可视化分析和故障诊断**。页面指出，Phoenix 的核心价值是从海量生产数据中发现问题、监控性能漂移并进行深度诊断。

Phoenix 的核心是：

> AI 可观测性。
> 

也就是把 RAG 系统内部的关键步骤都记录下来，例如：

```
用户输入
↓
检索过程
↓
检索结果
↓
Prompt 构造
↓
LLM 调用
↓
最终输出
```

工作流程：

```python
代码插桩 Instrumentation
↓
生成追踪数据 Traces
↓
启动 Phoenix Web 界面
↓
可视化查看完整执行链路
↓
筛选失败案例或低质量样本
↓
使用内置评估器 Evals 做进一步分析
```
