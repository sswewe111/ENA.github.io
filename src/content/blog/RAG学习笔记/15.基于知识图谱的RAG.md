# 基于知识图谱的RAG

传统 RAG 主要依赖文本块和向量相似度检索，但它不擅长理解实体之间的复杂关系；而知识图谱可以把实体、关系、属性显式组织起来，从而提升复杂查询、多跳推理、事实一致性和可解释性。

# 传统 RAG 的主要问题

| 问题 | 具体表现 | 造成的影响 | 举例理解 |
| --- | --- | --- | --- |
| 关系理解能力弱 | 传统 RAG 主要依赖向量相似度检索，只能找“语义相近”的文本块，不擅长理解实体之间的关系 | 对复杂关系型问题回答不稳定 | 用户问“公司 A 的母公司投资了哪个公司”，系统可能只检索到“公司 A 的介绍”，但找不到“母公司—投资对象”的关系链 |
| 上下文碎片化 | 文档通常会被切成多个 chunk（文本块），每个文本块相对独立 | 可能破坏原文中的逻辑关系和上下文连续性 | 一段内容前面讲“谁收购了谁”，后面讲“收购后的业务变化”，切分后模型可能只拿到一半信息 |
| 多跳推理能力有限 | 复杂问题往往需要经过多个实体和关系逐步推理，而传统 RAG 只是检索文本后交给模型总结 | 一旦某一步信息没有检索到，整个推理链就会断 | “A 的导师的学生创办了哪家公司”需要经过“A—导师—学生—公司”多跳关系 |
| 检索噪声较多 | 向量检索可能返回“看起来相关但不能支撑答案”的内容 | 容易干扰大模型判断，甚至引发幻觉 | 用户问“股权关系”，系统检索到“两个公司都属于新能源行业”，但这并不能回答股权问题 |
| 实体歧义问题 | 同一个名称可能对应多个实体，传统向量检索难以稳定区分 | 容易把不同实体的信息混在一起 | “Apple”可能指苹果公司、苹果水果，也可能指其他项目名称 |
| 别名和缩写处理能力弱 | 同一个实体可能有多个叫法，传统 RAG 不一定能自动统一 | 导致检索遗漏或信息不完整 | “OpenAI”“Open AI”“开放人工智能公司”可能被当成不同实体 |
| 可解释性较弱 | 传统 RAG 通常只返回文本依据，不一定能展示清晰的推理路径 | 用户难以判断答案是如何得出的 | 模型回答“公司 D 是投资对象”，但不能清楚说明是通过哪几层关系推出来的 |
| 事实一致性难保证 | 如果检索内容不完整或不准确，大模型可能根据自身知识补充 | 可能出现不基于材料的回答 | 检索材料没有提到某个投资时间，模型却自己补充了一个年份 |

# 知识图谱为什么能增强 RAG？

| 增强点 | 知识图谱的作用 | 对 RAG 的提升 | 举例理解 |
| --- | --- | --- | --- |
| 显式表示实体关系 | 知识图谱用“节点—边—属性”表达实体和关系 | 让系统不只是检索文本，而是能够直接查询关系 | `公司A --收购--> 公司B`，比普通文本更清晰 |
| 支持多跳推理 | 图结构天然适合沿着关系路径进行遍历 | 能解决复杂关系型问题 | `公司B ← 被收购 ← 公司A → 母公司 → 公司C → 投资 → 公司D` |
| 减少上下文碎片化 | 图谱把分散在不同文本中的实体和关系整合起来 | 即使原文分布在多个文档中，也能形成统一知识结构 | 多篇新闻分别提到“收购”“母公司”“投资”，图谱可以把它们连起来 |
| 提高检索准确性 | 检索对象从普通文本块变成实体、关系、路径和子图 | 检索结果更有针对性，噪声更少 | 用户问“股权关系”，系统优先检索“持股、控股、收购”等关系边 |
| 增强实体消歧能力 | 知识图谱可以为每个实体建立唯一 ID、别名、类型和属性 | 避免同名实体混淆 | “Apple”可以根据类型区分为“科技公司”或“水果” |
| 支持别名归一 | 可以把同一实体的不同名称映射到同一个节点 | 提高召回率，减少漏检 | “北大”“北京大学”“Peking University”都指向同一实体 |
| 提高事实可靠性 | 图谱中的关系通常带有来源、时间、置信度等信息 | 可以让模型基于结构化证据回答，降低幻觉 | `腾讯 --投资--> 某公司，时间：2019，来源：新闻A` |
| 增强可解释性 | 图谱可以展示答案对应的关系路径 | 用户能看到答案是如何推理出来的 | 答案不仅说“公司 D”，还能说明“因为公司 B 被 A 收购，A 的母公司 C 投资了 D” |
| 便于融合异构数据 | 知识图谱可以整合文本、表格、数据库、网页等多种来源 | 让 RAG 使用更完整的知识来源 | 企业内部文档、数据库、公告信息可以统一成图谱 |
| 适合复杂知识场景 | 对金融、医疗、法律、科研等关系密集型领域更友好 | 能处理普通 RAG 难以回答的复杂问题 | 金融风控中可以分析公司、股东、投资、交易、关联方关系 |

# GraphRAG 的核心变化

GraphRAG 检索的是：

```
实体
关系
路径
子图
图摘要
结构化证据
```

GraphRAG 不是简单地“多加一个知识图谱”，而是把 RAG 从“文本检索”推进到“知识利用”。

# GraphRAG 的通用流程

GraphRAG 的流程分成三个阶段：

```
知识图谱构建
↓
图谱检索
↓
增强生成
```

## 知识图谱构建

| 步骤 | 作用 |
| --- | --- |
| 知识抽取 | 从文本中抽取实体、关系、属性 |
| 实体归一 | 解决别名、缩写、同名实体问题 |
| 质量控制 | 检查三元组是否可靠，处理冲突信息 |
| 图谱融合 | 合并不同来源的数据，去重和对齐实体 |
| 存储索引 | 存入图数据库，并建立属性和关系索引 |

## 图谱检索

用户提出问题后，GraphRAG 不是简单做向量检索，而是先定位相关实体，再沿着图结构找证据。

典型流程是：

```
用户问题
↓
实体识别 / 实体链接
↓
找到图中的核心节点
↓
沿关系边扩展邻域
↓
发现路径或子图
↓
抽取结构化证据
```

## 增强生成

输入给大模型的内容不只是普通文本，而是：

```
用户问题
+
实体属性
+
关系路径
+
子图摘要
+
原文片段
+
来源信息
```

然后要求模型：

```
基于图证据回答
不要编造
必要时给出推理路径
```

# GraphRAG 的三类方法

## 知识驱动型

知识驱动型主要依赖知识图谱本身进行查询和推理。

流程大致是：

```
用户问题
↓
转换成图查询语句
↓
在知识图谱中查询
↓
返回实体、关系、路径
↓
生成答案
```

## 索引驱动型

索引驱动型不是完全依赖图数据库，而是把图谱结构信息融入文本索引中。

例如：

```
原始文本块
+
相邻实体
+
相关关系
+
子图摘要
↓
一起向量化
```

## 混合型

混合型同时使用：

```
图检索
+
文本检索
+
重排序
+
证据融合
```

它的逻辑是：

```
简单事实问题：优先查图
描述性问题：补充文本证据
复杂问题：先图上找路径，再根据路径节点检索相关文档
```

## 优缺点

| 方法类型 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- |
| 知识驱动型 | 关系查询准确；可解释性强；适合多跳推理；答案依据清晰；可以展示推理路径 | 依赖知识图谱质量；图谱不完整时容易查不到；构建和维护成本高；对实体识别、实体链接要求较高 | 金融风控、医疗问答、法律知识库、企业关系分析等强关系场景 |
| 索引驱动型 | 工程实现相对简单；容易接入现有 RAG 系统；不强依赖复杂图数据库查询；检索召回能力较好 | 显式推理能力弱；图结构利用不充分；可解释性不如知识驱动型；复杂多跳问题效果有限 | 普通知识库增强、企业文档问答、轻量级 GraphRAG 系统 |
| 混合型 | 综合效果最稳健；既能利用图结构关系，又能保留文本细节；适合复杂问答；能兼顾召回率和准确率 | 系统复杂度高；开发和调试难度大；成本较高；需要协调图数据库、向量数据库、重排序模型和大模型 | 复杂企业知识库、科研知识管理、金融分析、医疗辅助决策等高质量问答场景 |
